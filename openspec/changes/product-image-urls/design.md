## Context

`Product.image` and `ProductVariant.image` are `String` columns. Existing rows (from `prisma/seed.ts`) hold **absolute external URLs** (Platzi API, `picsum.photos`). New images uploaded through `POST /images` land in our RustFS bucket `bytes-products`, publicly readable at `https://s3.bstore.kroha.dev/bytes-products/<key>`. There is currently no rule for what the `image` string contains, so any code that bakes a full bucket URL into a row couples that row to today's domain + bucket name.

The bucket has a **public-read policy**, so no request signing is needed to display an image — resolving a key to a URL is pure string concatenation.

## Goals / Non-Goals

**Goals:**
- One unambiguous rule for what `image` stores, that keeps existing external URLs working untouched.
- Decouple owned-image references from the serving domain/bucket so a move is an env change, not a data migration.
- Centralize URL construction in the API so the web app and clients never assemble image URLs themselves.

**Non-Goals:**
- No DB schema change, no data migration of existing rows.
- No private/signed images (bucket is public-read; presigning is out of scope).
- No image transforms/thumbnails/CDN cache rules.

## Decisions

### Decision: Store a relative key for owned images; resolve to a URL on read

The `image` field stores either a **bare storage key** (`<uuid>.<ext>`, for images in our bucket) or an **absolute URL** (`http…`, for external/legacy images). On every read the API runs a resolver:

```
imageUrl(v) = v.startsWith('http') ? v : `${S3_PUBLIC_URL}/${S3_BUCKET}/${v}`
```

`POST /images` returns `{ key, url }` — callers persist `key`, and may use `url` for an instant preview.

**Why over alternatives:**
- **Store full bucket URL on the row** — simplest write path, but pins every row to one domain/bucket; renaming the bucket or moving off Cloudflare means rewriting every row. Rejected: portability is the whole point.
- **Always generate (key-only, no passthrough)** — cleanest model, but breaks the existing seed data full of external URLs; we'd have to re-host or rewrite all of it. Rejected: needless churn.
- **Presigned URLs per request** — required only for private objects; adds signing cost and makes URLs uncacheable/expiring. Rejected: bucket is public-read, so this buys nothing here.
- **Hybrid passthrough (chosen)** — absolute URLs pass through, keys get prefixed. Supports both worlds with a one-line resolver and zero migration.

### Decision: The `http` prefix is the discriminator

A value is treated as external iff it starts with `http`. Keys are generated by us (`crypto.randomUUID()` + sanitized extension) and never start with `http`, so the test is unambiguous. No separate "type" column needed.

## Risks / Trade-offs

- **A stored key collides with looking like a URL** → Mitigation: keys are UUID-based and can't start with `http`; the only URLs we ever store are genuinely external.
- **`S3_PUBLIC_URL` misconfigured → broken images site-wide** → Mitigation: it's validated at startup (`requireEnv`); a missing value fails fast instead of silently serving 404s.
- **Resolver forgotten on a new endpoint that returns `image`** → Mitigation: a single shared `imageUrl()` helper + a serializer applied wherever products/variants are returned, asserted by a test that a known key resolves and a known URL passes through.
- **Public base served before Cloudflare edge cert is ready** → Mitigation: orthogonal to this change; the resolver just emits the configured base, whatever it is (origin or CF domain).
